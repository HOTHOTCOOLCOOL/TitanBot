# Nanobot Project Context Architecture Redesign Plan

## 核心目标描述 (Goal Description)
随着 Nanobot 项目演进，根目录累积了超过 11 个重度长文 Markdown（如 [EVOLUTION.md](file:///d:/Python/nanobot/EVOLUTION.md), [progress_report.md](file:///d:/Python/nanobot/progress_report.md), [ARCHITECTURE_LESSONS.md](file:///d:/Python/nanobot/ARCHITECTURE_LESSONS.md) 等），总计几十至上百 KB。如果无差别喂给新会话的 Agent，将导致严重的 Token 浪费、“中间记忆丢失（Lost in the middle）”以及推理速度极烂。

本计划旨在打破“尽可能理解项目全貌”、“节约 Token”与“高复杂度需求”的“不可能三角”。方案核心是从**扁平的记忆模式**转向**分层结构的路由机制（Context Tiering）**。

## 空间分层架构设计 (Context Architecture)
我们将项目文档严格划分为三级：
*   **L0 (总控入口层 | < 3KB)**：仅仅一个文件（[.agent/CODEBASE_MAP.md](file:///d:/Python/nanobot/.agent/CODEBASE_MAP.md)），充当类似“路由器”和“目录地图”的作用。它是新会话中 Agent **唯一**需要在一开始通读的文件。
*   **L1 (按需参考层 | 规则与现状)**：如 [SECURITY.md](file:///d:/Python/nanobot/SECURITY.md), [TOOLS.md](file:///d:/Python/nanobot/TOOLS.md), 裁剪后的 [progress_report.md](file:///d:/Python/nanobot/progress_report.md)。Agent 只有在被 L0 路由引导，或者明确要做相关任务时才通过工具 `view_file` 单独读取。
*   **L2 (冷库归档层 | 历史与审计)**：过去的架构长文、测试清单、历代演进史。默认存放在 `docs/archive/`，对新会话的 Agent *绝对干预阻断/不可见*，除非人类显式要求查阅历史。

### 2. 文档瘦身与截断过滤 (Phase 2: Document "Truncation" & Slicing)

#### [MODIFY] ARCHITECTURE_LESSONS.md & LESSONS_LEARNED.md

### 3. 建立“总路由器” (Phase 3: The Single Router L0)
彻底强化地图中心。

#### [MODIFY] .agent/CODEBASE_MAP.md
- **动作**：精简重写。不再罗列细枝末节的代码逻辑。

## 自动化执行指引 (Agent Guidelines for Future Sessions)
在完成以上文档结构的重组后，**未来开启的任何新会话，如何精简地告诉新 Agent 该架构并让它遵守？**

你甚至不需要给 Agent 写长篇大论，只需要在每一次新的提问前（或通过 Agent 系统的 System Prompt / Custom Instructions 固化）附带如下一段**核心咒语（Invocation Prompt）**：

```text
【系统强制指令 / SYSTEM CRITICAL INSTRUCTION】
你正在参与 Nanobot 项目的开发。由于本项目庞大，请严格遵循“分层路由上下文（Context Tiering Architecture）”获取信息，避免 Token 浪费：
1. **禁止全盘读取**：绝对禁止直接列出并读取文件树中的所有 Markdown 文件。
2. **唯一入口 L0**：你当前唯一被允许率先读取的文件是 `.agent/CODEBASE_MAP.md`。
3. **按需路由 L1**：阅读 MAP 后，请根据其中的路由指针（Pointers），只按需读取你当前解决问题所必需的特定 L1 文档（如 ` progress_report.md` 或相关的 Rules）。历史文件 (docs/archive) 永远不要读取。
```

当把这段话固化到新会话里，Agent 的智能就会自我驱动去“按图索骥”，彻底打破那个不可能三角。
