# AI Context Tiering Architecture (分层路由架构) 初始化模板

在启动任何新的、预期会长期演进的 AI 辅助项目时，请按照以下三个步骤初始化你的“上下文架构”。这将极大减少 AI 阅读冗余文档带来的 Token 消耗，并彻底终结 AI 的“上下文迷失”与幻觉问题。

---

## 阶段一：建立“规则护城河” (The Rule Engine)

在新项目根目录下，直接创建 AI 编辑器的全局指令文件（例如 `.cursorrules`, `.clinerules`, 或 `.windsurfrules`）。

将下方的 `[强制指令]` 原封不动地复制进去：

```markdown
【系统强制指令 / SYSTEM CRITICAL INSTRUCTION】
你正在参与本项目的开发。为保障系统稳定性与上下文窗口效率，请严格遵循“分层路由上下文（Context Tiering Architecture）”获取信息，避免 Token 浪费：
1. **禁止全盘读取**：绝对禁止直接自动搜索、列出并读取项目树中的所有 Markdown 文档。
2. **唯一入口 L0**：每轮新会话，你当前唯一被允许率先读取的文件是 `.agent/CODEBASE_MAP.md`。
3. **按需路由 L1**：阅读 L0 架构图后，请严格根据其中的路由指针（Pointers），只按需读取你当前解决该特定问题所必需的 L1 状态文档（如进度报告、设计规范）。历史存档文档永远不要读取。
```

---

## 阶段二：建立 L0 入口地图 (The L0 Map)

在项目根目录下创建一个隐藏文件夹 `.agent/`，并在其中创建 `CODEBASE_MAP.md`。
这是所有 AI 代理在项目中永远的**第一个也是唯一一个**着陆点。

**`.agent/CODEBASE_MAP.md` 的模板如下：**

```markdown
# 项目全景图 & AI 路由节点 (CODEBASE_MAP)

> **当前状态**: [开发中 / 维护中] | **核心目标**: [用一句话描述项目当前阶段的核心目标]

## 1. 核心路由指针 (L1 文档)
当且仅当你需要了解以下领域的详细信息时，才允许你去读取对应的文件：

- 📍 **当前进度与任务**: 详见 `docs/progress_report.md` (包含待办事项和最近完成的里程碑)
- 📍 **架构与开发规范**: 详见 `docs/rules/ARCHITECTURE.md` (包含代码规范、技术栈设计)
- 📍 **测试与质量追踪**: 详见 `docs/TEST_TRACKER.md` (包含测试覆盖率与已知 Bug 列表)

## 2. 核心模块代码映射图表
| 模块名称 | 路径入口 | 核心职责描述 |
| --- | --- | --- |
| 核心引擎 | `src/engine/` | 处理底层的逻辑运算与初始化 |
| 用户接口 | `src/ui/` | 前端渲染与用户交互拦截 |
| 存储层 | `src/storage/` | 数据库交互与状态持久化定义 |

## 3. 注意事项 (Anti-patterns)
- **永远不要** 读取 `docs/archive/` 目录中的任何旧文档。
- 只有在执行完相应的开发任务后，才被允许同时更新对应的 L1 文档。
```

---

## 阶段三：初始化 L1 状态文档 (The L1 Documents)

在项目初期，你只需要初始化最基础的一个或两个 L1 文档，不需要写得很繁琐。
随着项目生长，你可以让 AI 慢慢帮你补充 L1 文档：

1. 创建 `docs/progress_report.md`：让 AI 记录最近做了什么，下一步要做什么。
2. 创建 `docs/rules/ARCHITECTURE.md`（可选）：定义技术选型（比如强制使用 Python 3.12，或强制前端使用 Tailwind CSS 等）。

---

## 🚦 它是如何工作的？(运转逻辑)
当你在新项目中第一次提出要求：*"帮我修一个 UI 层的 Bug"*
1. AI 接到任务，后台触发 `.cursorrules`。
2. AI 第一步被迫去读 `.agent/CODEBASE_MAP.md` (L0)。
3. AI 从 L0 中得知：代码在 `src/ui/`，且当前的全局进度和已知库在 `docs/progress_report.md` (L1)。
4. AI 仅读取必需的 UI 代码和相关的进度报告。
5. AI 精确修复 Bug。这就避免了它去读后端存储代码，或去读一堆无关的历史调研记录。
